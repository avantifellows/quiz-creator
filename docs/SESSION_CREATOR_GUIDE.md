# Session Creator Guide

## Purpose

This document explains, in product terms, what each field in `quiz-creator` means, what options are available, and how those choices affect the session that gets created.

It is written mainly for quiz sessions, because that is where most of the metadata lives, but it also covers live-session behavior where relevant.

## What A Session Is

A session is the access wrapper around a destination:

- For `quiz`, the session controls who can enter, when they can enter, what portal flow they see first, and how the quiz behaves.
- For live platforms like `youtube`, `meet`, `zoom`, `AF-plio`, `SCERT-plio`, and `others`, the session controls the same entry/auth flow, but redirects users to the live destination instead of a quiz.

In practice, session creation has 3 steps:

1. `Basic Details`
2. `Platform Details`
3. `Timeline Details`

Some fields are chosen by the user. Some are generated by downstream systems after save.

## Where The Dropdown Values Come From

There are 2 kinds of dropdowns in `quiz-creator`.

### Fixed dropdowns in code

These are hardcoded in the app:

- Platform
- Grade
- Session Type
- Auth Type
- Course
- Stream
- Test Format
- Test Purpose
- Test Type
- Quiz Display Format
- Optional Limit
- Session Pattern
- Active Days

### Dynamic dropdowns from backend data

These are fetched live:

- `Group`: from `/auth-group`
- `Quiz Batch` and `Class Batch`: from `/batch`
- `Signup Form Name` and `Popup Form Name`: from `/form-schema`

Form options are further filtered by form name:

- signup forms are the form schemas whose name contains `Registration`
- popup forms are the form schemas whose name contains `Profile`

They are also filtered by group short-name matching, for example `Delhi`, `Punjab`, `TN`, `Enable`.

### Basic step behavior rules

- If `Platform = quiz`, `Quiz Batch` is required.
- If `Platform != quiz`, `Class Batch` is required instead.
- If `Platform != quiz`, the session appears under the `Live Classes` table in quiz-creator.
- If `Platform = no-platform`, redirection is forcibly turned off.
- If `redirection = false`, quiz/live platform details are skipped and cleared, and the user flow goes directly from basic details to timeline.
- If `Session Type = sign-up`, `Display sign up form` is automatically turned on.

### Batch filtering rules

The batch dropdowns are not a raw full list. They are filtered by group and by platform:

- quiz sessions use parent batches
- live sessions use child batches

There are also special mappings for school groups:

- `TNSchools` uses `TNStudents` batches
- `GujaratSchools` uses `GujaratStudents` batches
- `PunjabSchools` uses `PunjabStudents` batches
- `EnableSchools` uses `EnableStudents` batches

### Group-based presets

The form auto-fills some defaults for certain groups:

| Group                                                                          | Auto behavior                                                                               |
| ------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------- |
| `EnableStudents`                                                               | Defaults to `sign-in`, auth type `ID,DOB`, sign-up form enabled, ID generation enabled      |
| `DelhiStudents`, `GujaratStudents`, `HimachalStudents`, `FeedingIndiaStudents` | Defaults to `sign-in`, auth type `ID`, no sign-up form, no popup form, no ID generation     |
| `UttarakhandStudents`                                                          | Defaults to `sign-in`, auth type `ID,DOB`, no sign-up form, no popup form, no ID generation |
| `HaryanaStudents` and most other groups                                        | Clears forms and keeps ID generation off unless user changes it                             |

### Important quiz behavior

#### `Test Type`

This is the most important quiz setting because it changes the experience:

- `assessment`: standard quiz flow
- `homework`: also a quiz flow, but current creator logic forces simpler marking
- `omr-assessment`: quiz can be opened in OMR mode
- `form`: questionnaire/form flow instead of a normal quiz

#### Special behavior when `Test Type = form`

When `form` is selected, quiz-creator auto-applies these defaults:

- `Test Format = questionnaire`
- `Quiz Display Format = qa`
- `Optional Limit = N/A`
- `Show Answers = true`
- `Show Scores = false`
- `Shuffle = false`
- `Stream = Others`

It also changes the UI:

- `CMS URL` becomes `Google Sheets Link`
- `Sheet Name` appears
- `Single Page Header Text` appears

Downstream behavior for forms:

- the ETL/session-creation flow forces `single_page_mode = Yes`
- portal redirects users to the `form` route, not the regular quiz route
- portal adds `singlePageMode=true` and `autoStart=true`

#### `Quiz Display Format (gurukul_format_type)`

This field is used by the Gurukul app.

Behavior on Gurukul:

- `qa`: show only the normal `Start Test` button
- `omr`: show only the `Fill OMR` button
- `both`: show both buttons

So this is not just passive metadata. It controls which launch options are visible in Gurukul for the session.

#### `Show Answers`

This maps to quiz backend `review_immediate`.

- If `Show Answers = true`, users can review answers immediately after finishing.
- If `Show Answers = false`, answer review is delayed until the session end time has passed.

#### `Show Scores`

This controls whether the scorecard shows marks and detailed metrics.

- If `true`, the scorecard shows performance details.
- If `false`, users mostly get a completion message instead of score details.
- For forms, the creator defaults this to `false`.

#### `Shuffle Questions`

This is used by quiz session creation in the quiz backend.

- If `true`, question order is shuffled.
- The backend currently shuffles within question buckets/blocks, not as one fully random flat list.
- OMR mode does not use the shuffled order in the same way as normal quiz mode.

#### `Optional Limit`

This affects how many questions can be attempted from each question set after ETL creates the quiz.

Current mappings in the session creator flow:

- `JEE`: applies a JEE-style optional-attempt structure
- `NEET`: applies a NEET-style optional-attempt structure
- `CUET`: applies a CUET-style optional-attempt structure
- `N/A` or `NA`: no special optional-limit logic beyond full-attempt defaults

This is meaningful quiz behavior, not just a label.

#### `Marking Scheme`

This exists in session JSON as `meta_data.marking_scheme`, but in the current quiz-creator UI it is not actually user-configurable.

Current creator behavior:

- `homework` and `form` are stored as `1, 0`
- everything else is stored as `4,-1`

The ETL flow then converts that into the quiz/question-set scoring logic used by quiz-backend.

#### `Test Purpose`

This is stored and carried downstream, and it is also present in sheet/import flows.

What is clear:

- it is saved as `meta_data.test_purpose`
- it is useful as a business/reporting label

What is not clearly confirmed in the code reviewed:

- a direct runtime behavior in portal, quiz frontend, or Gurukul that changes purely because of `test_purpose`

So it should currently be treated as a categorization field unless a team-specific reporting process depends on it.

## Platform Details For Live Sessions

If `Platform != quiz` and redirection stays on, the form switches to live-session fields.

| UI field        | Meaning                                    | Behavior                                               |
| --------------- | ------------------------------------------ | ------------------------------------------------------ |
| `Platform Link` | Actual destination URL                     | Saved to `platform_link`                               |
| `Platform ID`   | Extracted or manual destination identifier | Saved to `platform_id`                                 |
| `Subject`       | Subject tags for the class                 | Saved to `meta_data.subject` as comma-separated values |

### Live-session link behavior

`quiz-creator` tries to auto-extract `platform_id` from the URL for common platforms:

- Google Meet
- YouTube
- Zoom
- Plio
- Google Drive links

If the platform is `others` and no ID can be extracted, it generates a timestamp-like fallback ID.

### Live-session purpose behavior

For non-quiz sessions, downstream ETL derives `purpose.sub-type` mainly from platform and subject:

- `registration`
- `plio`
- `liveclass`
- `scholarship`
- `mentorship`
- `experiment`

That `purpose` value is mostly system-derived, not manually chosen in quiz-creator.

## Timeline Details

| UI field                      | Options / format       | Meaning                      | Main behavior                          |
| ----------------------------- | ---------------------- | ---------------------------- | -------------------------------------- |
| `Start Date` + `Start Time`   | Datetime               | When availability starts     | Saved to `start_time`                  |
| `End Date` + `End Time`       | Datetime               | When availability ends       | Saved to `end_time`                    |
| `Expected Session Attendance` | Number                 | Expected number of users     | Saved to `meta_data.test_takers_count` |
| `Session Pattern`             | `continuous`, `weekly` | Availability model           | Saved to `repeat_schedule.type`        |
| `Days Active`                 | Monday-Sunday          | Which weekdays are active    | Saved to `repeat_schedule.params`      |
| `Is Enabled?`                 | On/off                 | Master session on/off switch | Saved to `is_active`                   |

### Session pattern behavior

#### `continuous`

This means one continuous availability window from start datetime to end datetime.

Downstream effect:

- one session occurrence is created spanning the full time window

Typical use:

- quiz sessions
- forms
- some sessionless/no-platform flows

#### `weekly`

This means recurring availability on selected weekdays between the chosen start and end dates.

Downstream effect:

- multiple session occurrences are generated, one for each selected weekday until the end date

Typical use:

- recurring live classes

### `Is Enabled?`

This is the practical open/close switch.

Portal behavior:

- even if a valid occurrence exists, the portal still checks `is_active`
- if `is_active = false`, the session is treated as closed

## How The Example Session Should Be Read

Using the sample quiz session JSON you shared:

- `platform = quiz` means the session ultimately launches a quiz
- `platform_id` and `platform_link` both point to the quiz ID in this flow
- `portal_link` is the auth entry URL, not the quiz URL itself
- `session_id` is the public session token used by portal and Gurukul
- `auth_type = ID` means portal asks for user ID only
- `type = sign-in` means users are expected to sign in, not register first
- `redirection = true` means portal should send the user onward after auth
- `repeat_schedule.type = continuous` means the quiz is available continuously between `start_time` and `end_time`
- `gurukul_format_type = qa` means Gurukul should show only the normal start-test button, not the OMR button
- `show_scores = false` means users should not see score details on completion
- `show_answers = false` means immediate answer review is blocked until the session end time passes
- `updated_keys` is a system patch-tracking field, not something selected in the form

## Practical Summary

If you only need the shortest mental model, these are the most important levers:

- `Platform` decides where the user ends up.
- `Group`, `Batch`, `Auth Type`, `Signup Form`, `Popup Form`, and `ID Generation` decide the entry and identity flow.
- `Test Type`, `Show Answers`, `Show Scores`, `Shuffle`, `Optional Limit`, and `Quiz Display Format` decide the learner-facing quiz behavior.
- `Session Pattern`, dates, and `Is Enabled?` decide when the session is actually open.

The most important fields for downstream behavior are:

- `platform`
- `type`
- `auth_type`
- `signup_form`
- `popup_form`
- `id_generation`
- `redirection`
- `test_type`
- `gurukul_format_type`
- `show_answers`
- `show_scores`
- `shuffle`
- `optional_limits`
- `repeat_schedule`
- `is_active`
